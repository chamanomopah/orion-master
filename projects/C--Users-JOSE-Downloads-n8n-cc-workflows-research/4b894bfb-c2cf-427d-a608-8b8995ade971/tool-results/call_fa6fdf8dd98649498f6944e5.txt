   245→## Installing MCP servers
   246→
   247→MCP servers can be configured in three different ways depending on your needs:
   248→
   249→### Option 1: Add a remote HTTP server
   250→
   251→HTTP servers are the recommended option for connecting to remote MCP servers. This is the most widely supported transport for cloud-based services.
   252→
   253→```bash  theme={null}
   254→# Basic syntax
   255→claude mcp add --transport http <name> <url>
   256→
   257→# Real example: Connect to Notion
   258→claude mcp add --transport http notion https://mcp.notion.com/mcp
   259→
   260→# Example with Bearer token
   261→claude mcp add --transport http secure-api https://api.example.com/mcp \
   262→  --header "Authorization: Bearer your-token"
   263→```
   264→
   265→### Option 2: Add a remote SSE server
   266→
   267→<Warning>
   268→  The SSE (Server-Sent Events) transport is deprecated. Use HTTP servers instead, where available.
   269→</Warning>
   270→
   271→```bash  theme={null}
   272→# Basic syntax
   273→claude mcp add --transport sse <name> <url>
   274→
   275→# Real example: Connect to Asana
   276→claude mcp add --transport sse asana https://mcp.asana.com/sse
   277→
   278→# Example with authentication header
   279→claude mcp add --transport sse private-api https://api.company.com/sse \
   280→  --header "X-API-Key: your-key-here"
   281→```
   282→
   283→### Option 3: Add a local stdio server
   284→
   285→Stdio servers run as local processes on your machine. They're ideal for tools that need direct system access or custom scripts.
   286→
   287→```bash  theme={null}
   288→# Basic syntax
   289→claude mcp add [options] <name> -- <command> [args...]
   290→
   291→# Real example: Add Airtable server
   292→claude mcp add --transport stdio --env AIRTABLE_API_KEY=YOUR_KEY airtable \
   293→  -- npx -y airtable-mcp-server
   294→```
   295→
   296→<Note>
   297→  **Important: Option ordering**
   298→
   299→  All options (`--transport`, `--env`, `--scope`, `--header`) must come **before** the server name. The `--` (double dash) then separates the server name from the command and arguments that get passed to the MCP server.
   300→
   301→  For example:
   302→
   303→  * `claude mcp add --transport stdio myserver -- npx server` → runs `npx server`
   304→  * `claude mcp add --transport stdio --env KEY=value myserver -- python server.py --port 8080` → runs `python server.py --port 8080` with `KEY=value` in environment
   305→
   306→  This prevents conflicts between Claude's flags and the server's flags.
   307→</Note>
   308→
   309→### Managing your servers
   310→
   311→Once configured, you can manage your MCP servers with these commands:
   312→
   313→```bash  theme={null}
   314→# List all configured servers
   315→claude mcp list
   316→
   317→# Get details for a specific server
   318→claude mcp get github
   319→
   320→# Remove a server
   321→claude mcp remove github
   322→
   323→# (within Claude Code) Check server status
   324→/mcp
   325→```

   327→### Dynamic tool updates
   328→
   329→Claude Code supports MCP `list_changed` notifications, allowing MCP servers to dynamically update their available tools, prompts, and resources without requiring you to disconnect and reconnect. When an MCP server sends a `list_changed` notification, Claude Code automatically refreshes the available capabilities from that server.
   330→
   331→<Tip>
   332→  Tips:
   333→
   334→  * Use the `--scope` flag to specify where the configuration is stored:
   335→    * `local` (default): Available only to you in the current project (was called `project` in older versions)
   336→    * `project`: Shared with everyone in the project via `.mcp.json` file
   337→    * `user`: Available to you across all projects (was called `global` in older versions)
   338→  * Set environment variables with `--env` flags (for example, `--env KEY=value`)
   339→  * Configure MCP server startup timeout using the MCP\_TIMEOUT environment variable (for example, `MCP_TIMEOUT=10000 claude` sets a 10-second timeout)
   340→  * Claude Code will display a warning when MCP tool output exceeds 10,000 tokens. To increase this limit, set the `MAX_MCP_OUTPUT_TOKENS` environment variable (for example, `MAX_MCP_OUTPUT_TOKENS=50000`)
   341→  * Use `/mcp` to authenticate with remote servers that require OAuth 2.0 authentication
   342→</Tip>
   343→
   344→<Warning>
   345→  **Windows Users**: On native Windows (not WSL), local MCP servers that use `npx` require the `cmd /c` wrapper to ensure proper execution.
   346→
   347→  ```bash  theme={null}
   348→  # This creates command="cmd" which Windows can execute
   349→  claude mcp add --transport stdio my-server -- cmd /c npx -y @some/package
   350→  ```
   351→
   352→  Without the `cmd /c` wrapper, you'll encounter "Connection closed" errors because Windows cannot directly execute `npx`. (See the note above for an explanation of the `--` parameter.)
   353→</Warning>

   355→### Plugin-provided MCP servers
   356→
   357→[Plugins](/en/plugins) can bundle MCP servers, automatically providing tools and integrations when the plugin is enabled. Plugin MCP servers work identically to user-configured servers.
   358→
   359→**How plugin MCP servers work**:
   360→
   361→* Plugins define MCP servers in `.mcp.json` at the plugin root or inline in `plugin.json`
   362→* When a plugin is enabled, its MCP servers start automatically
   363→* Plugin MCP tools appear alongside manually configured MCP tools
   364→* Plugin servers are managed through plugin installation (not `/mcp` commands)
   365→
   366→**Example plugin MCP configuration**:
   367→
   368→In `.mcp.json` at plugin root:
   369→
   370→```json  theme={null}
   371→{
   372→  "database-tools": {
   373→    "command": "${CLAUDE_PLUGIN_ROOT}/servers/db-server",
   374→    "args": ["--config", "${CLAUDE_PLUGIN_ROOT}/config.json"],
   375→    "env": {
   376→      "DB_URL": "${DB_URL}"
   377→    }
   378→  }
   379→}
   380→```
   381→
   382→Or inline in `plugin.json`:
   383→
   384→```json  theme={null}
   385→{
   386→  "name": "my-plugin",
   387→  "mcpServers": {
   388→    "plugin-api": {
   389→      "command": "${CLAUDE_PLUGIN_ROOT}/servers/api-server",
   390→      "args": ["--port", "8080"]
   391→    }
   392→  }
   393→}
   394→```
   395→
   396→**Plugin MCP features**:
   397→
   398→* **Automatic lifecycle**: Servers start when plugin enables, but you must restart Claude Code to apply MCP server changes (enabling or disabling)
   399→* **Environment variables**: Use `${CLAUDE_PLUGIN_ROOT}` for plugin-relative paths
   400→* **User environment access**: Access to same environment variables as manually configured servers
   401→* **Multiple transport types**: Support stdio, SSE, and HTTP transports (transport support may vary by server)
   402→
   403→**Viewing plugin MCP servers**:
   404→
   405→```bash  theme={null}
   406→# Within Claude Code, see all MCP servers including plugin ones
   407→/mcp
   408→```
   409→
   410→Plugin servers appear in the list with indicators showing they come from plugins.
   411→
   412→**Benefits of plugin MCP servers**:
   413→
   414→* **Bundled distribution**: Tools and servers packaged together
   415→* **Automatic setup**: No manual MCP configuration needed
   416→* **Team consistency**: Everyone gets the same tools when plugin is installed
   417→
   418→See the [plugin components reference](/en/plugins-reference#mcp-servers) for details on bundling MCP servers with plugins.

   420→## MCP installation scopes
   421→
   422→MCP servers can be configured at three different scope levels, each serving distinct purposes for managing server accessibility and sharing. Understanding these scopes helps you determine the best way to configure servers for your specific needs.
   423→
   424→### Local scope
   425→
   426→Local-scoped servers represent the default configuration level and are stored in `~/.claude.json` under your project's path. These servers remain private to you and are only accessible when working within the current project directory. This scope is ideal for personal development servers, experimental configurations, or servers containing sensitive credentials that shouldn't be shared.
   427→
   428→<Note>
   429→  The term "local scope" for MCP servers differs from general local settings. MCP local-scoped servers are stored in `~/.claude.json` (your home directory), while general local settings use `.claude/settings.local.json` (in the project directory). See [Settings](/en/settings#settings-files) for details on settings file locations.
   430→</Note>
   431→
   432→```bash  theme={null}
   433→# Add a local-scoped server (default)
   434→claude mcp add --transport http stripe https://mcp.stripe.com
   435→
   436→# Explicitly specify local scope
   437→claude mcp add --transport http stripe --scope local https://mcp.stripe.com
   438→```
   439→
   440→### Project scope
   441→
   442→Project-scoped servers enable team collaboration by storing configurations in a `.mcp.json` file at your project's root directory. This file is designed to be checked into version control, ensuring all team members have access to the same MCP tools and services. When you add a project-scoped server, Claude Code automatically creates or updates this file with the appropriate configuration structure.
   443→
   444→```bash  theme={null}
   445→# Add a project-scoped server
   446→claude mcp add --transport http paypal --scope project https://mcp.paypal.com/mcp
   447→```
   448→
   449→The resulting `.mcp.json` file follows a standardized format:
   450→
   451→```json  theme={null}
   452→{
   453→  "mcpServers": {
   454→    "shared-server": {
   455→      "command": "/path/to/server",
   456→      "args": [],
   457→      "env": {}
   458→    }
   459→  }
   460→}
   461→```
   462→
   463→For security reasons, Claude Code prompts for approval before using project-scoped servers from `.mcp.json` files. If you need to reset these approval choices, use the `claude mcp reset-project-choices` command.
   464→
   465→### User scope
   466→
   467→User-scoped servers are stored in `~/.claude.json` and provide cross-project accessibility, making them available across all projects on your machine while remaining private to your user account. This scope works well for personal utility servers, development tools, or services you frequently use across different projects.
   468→
   469→```bash  theme={null}
   470→# Add a user server
   471→claude mcp add --transport http hubspot --scope user https://mcp.hubspot.com/anthropic
   472→```
   473→
   474→### Choosing the right scope
   475→
   476→Select your scope based on:
   477→
   478→* **Local scope**: Personal servers, experimental configurations, or sensitive credentials specific to one project
   479→* **Project scope**: Team-shared servers, project-specific tools, or services required for collaboration
   480→* **User scope**: Personal utilities needed across multiple projects, development tools, or frequently used services
   481→
   482→<Note>
   483→  **Where are MCP servers stored?**
   484→
   485→  * **User and local scope**: `~/.claude.json` (in the `mcpServers` field or under project paths)
   486→  * **Project scope**: `.mcp.json` in your project root (checked into source control)
   487→  * **Managed**: `managed-mcp.json` in system directories (see [Managed MCP configuration](#managed-mcp-configuration))
   488→</Note>
   489→
   490→### Scope hierarchy and precedence
   491→
   492→MCP server configurations follow a clear precedence hierarchy. When servers with the same name exist at multiple scopes, the system resolves conflicts by prioritizing local-scoped servers first, followed by project-scoped servers, and finally user-scoped servers. This design ensures that personal configurations can override shared ones when needed.
   493→
   494→### Environment variable expansion in `.mcp.json`
   495→
   496→Claude Code supports environment variable expansion in `.mcp.json` files, allowing teams to share configurations while maintaining flexibility for machine-specific paths and sensitive values like API keys.
   497→
   498→**Supported syntax:**
   499→
   500→* `${VAR}` - Expands to the value of environment variable `VAR`
   501→* `${VAR:-default}` - Expands to `VAR` if set, otherwise uses `default`
   502→
   503→**Expansion locations:**
   504→Environment variables can be expanded in:
   505→
   506→* `command` - The server executable path
   507→* `args` - Command-line arguments
   508→* `env` - Environment variables passed to the server
   509→* `url` - For HTTP server types
   510→* `headers` - For HTTP server authentication
   511→
   512→**Example with variable expansion:**
   513→
   514→```json  theme={null}
   515→{
   516→  "mcpServers": {
   517→    "api-server": {
   518→      "type": "http",
   519→      "url": "${API_BASE_URL:-https://api.example.com}/mcp",
   520→      "headers": {
   521→        "Authorization": "Bearer ${API_KEY}"
   522→      }
   523→    }
   524→  }
   525→}
   526→```
   527→
   528→If a required environment variable is not set and has no default value, Claude Code will fail to parse the config.

   654→  * OAuth authentication works with HTTP servers

   656→
   657→### Use a fixed OAuth callback port
   658→
   659→Some MCP servers require a specific redirect URI registered in advance. By default, Claude Code picks a random available port for the OAuth callback. Use `--callback-port` to fix the port so it matches a pre-registered redirect URI of the form `http://localhost:PORT/callback`.
   660→
   661→You can use `--callback-port` on its own (with dynamic client registration) or together with `--client-id` (with pre-configured credentials).
   662→
   663→```bash  theme={null}
   664→# Fixed callback port with dynamic client registration
   665→claude mcp add --transport http \
   666→  --callback-port 8080 \
   667→  my-server https://mcp.example.com/mcp
   668→```
   669→
   670→### Use pre-configured OAuth credentials
   671→
   672→Some MCP servers don't support automatic OAuth setup. If you see an error like "Incompatible auth server: does not support dynamic client registration," the server requires pre-configured credentials. Register an OAuth app through the server's developer portal first, then provide the credentials when adding the server.
   673→
   674→<Steps>
   675→  <Step title="Register an OAuth app with the server">
   676→    Create an app through the server's developer portal and note your client ID and client secret.
   677→
   678→    Many servers also require a redirect URI. If so, choose a port and register a redirect URI in the format `http://localhost:PORT/callback`. Use that same port with `--callback-port` in the next step.
   679→  </Step>
   680→
   681→  <Step title="Add the server with your credentials">
   682→    Choose one of the following methods. The port used for `--callback-port` can be any available port. It just needs to match the redirect URI you registered in the previous step.
   683→
   684→    <Tabs>
   685→      <Tab title="claude mcp add">
   686→        Use `--client-id` to pass your app's client ID. The `--client-secret` flag prompts for the secret with masked input:
   687→
   688→        ```bash  theme={null}
   689→        claude mcp add --transport http \
   690→          --client-id your-client-id --client-secret --callback-port 8080 \
   691→          my-server https://mcp.example.com/mcp
   692→        ```
   693→      </Tab>
   694→
   695→      <Tab title="claude mcp add-json">
   696→        Include the `oauth` object in the JSON config and pass `--client-secret` as a separate flag:
   697→
   698→        ```bash  theme={null}
   699→        claude mcp add-json my-server \
   700→          '{"type":"http","url":"https://mcp.example.com/mcp","oauth":{"clientId":"your-client-id","callbackPort":8080}}' \
   701→          --client-secret
   702→        ```
   703→      </Tab>
   704→
   705→      <Tab title="claude mcp add-json (callback port only)">
   706→        Use `--callback-port` without a client ID to fix the port while using dynamic client registration:
   707→
   708→        ```bash  theme={null}
   709→        claude mcp add-json my-server \
   710→          '{"type":"http","url":"https://mcp.example.com/mcp","oauth":{"callbackPort":8080}}'
   711→        ```
   712→      </Tab>
   713→
   714→      <Tab title="CI / env var">
   715→        Set the secret via environment variable to skip the interactive prompt:
   716→
   717→        ```bash  theme={null}
   718→        MCP_CLIENT_SECRET=your-secret claude mcp add --transport http \
   719→          --client-id your-client-id --client-secret --callback-port 8080 \
   720→          my-server https://mcp.example.com/mcp
   721→        ```
   722→      </Tab>
   723→    </Tabs>
   724→  </Step>
   725→
   726→  <Step title="Authenticate in Claude Code">
   727→    Run `/mcp` in Claude Code and follow the browser login flow.
   728→  </Step>
   729→</Steps>
   730→
   731→<Tip>
   732→  Tips:
   733→
   734→  * The client secret is stored securely in your system keychain (macOS) or a credentials file, not in your config
   735→  * If the server uses a public OAuth client with no secret, use only `--client-id` without `--client-secret`
   736→  * `--callback-port` can be used with or without `--client-id`
   737→  * These flags only apply to HTTP and SSE transports. They have no effect on stdio servers
   738→  * Use `claude mcp get <name>` to verify that OAuth credentials are configured for a server
   739→</Tip>
   740→
   741→### Override OAuth metadata discovery
   742→
   743→If your MCP server returns errors on the standard OAuth metadata endpoint (`/.well-known/oauth-authorization-server`) but exposes a working OIDC endpoint, you can tell Claude Code to fetch OAuth metadata directly from a URL you specify, bypassing the standard discovery chain.
   744→
   745→Set `authServerMetadataUrl` in the `oauth` object of your server's config in `.mcp.json`:
   746→
   747→```json  theme={null}
   748→{
   749→  "mcpServers": {
   750→    "my-server": {
   751→      "type": "http",
   752→      "url": "https://mcp.example.com/mcp",
   753→      "oauth": {
   754→        "authServerMetadataUrl": "https://auth.example.com/.well-known/openid-configuration"
   755→      }
   756→    }
   757→  }
   758→}
   759→```
   760→
   761→The URL must use `https://`. This option requires Claude Code v2.1.64 or later.

   921→## MCP output limits and warnings
   922→
   923→When MCP tools produce large outputs, Claude Code helps manage the token usage to prevent overwhelming your conversation context:
   924→
   925→* **Output warning threshold**: Claude Code displays a warning when any MCP tool output exceeds 10,000 tokens
   926→* **Configurable limit**: You can adjust the maximum allowed MCP output tokens using the `MAX_MCP_OUTPUT_TOKENS` environment variable
   927→* **Default limit**: The default maximum is 25,000 tokens
   928→
   929→To increase the limit for tools that produce large outputs:
   930→
   931→```bash  theme={null}
   932→# Set a higher limit for MCP tool outputs
   933→export MAX_MCP_OUTPUT_TOKENS=50000
   934→claude
   935→```
   936→
   937→This is particularly useful when working with MCP servers that:
   938→
   939→* Query large datasets or databases
   940→* Generate detailed reports or documentation
   941→* Process extensive log files or debugging information
   942→
   943→<Warning>
   944→  If you frequently encounter output warnings with specific MCP servers, consider increasing the limit or configuring the server to paginate or filter its responses.
   945→</Warning>

   947→## Respond to MCP elicitation requests
   948→
   949→MCP servers can request structured input from you mid-task using elicitation. When a server needs information it can't get on its own, Claude Code displays an interactive dialog and passes your response back to the server. No configuration is required on your side: elicitation dialogs appear automatically when a server requests them.
   950→
   951→Servers can request input in two ways:
   952→
   953→* **Form mode**: Claude Code shows a dialog with form fields defined by the server (for example, a username and password prompt). Fill in the fields and submit.
   954→* **URL mode**: Claude Code opens a browser URL for authentication or approval. Complete the flow in the browser, then confirm in the CLI.
   955→
   956→To auto-respond to elicitation requests without showing a dialog, use the [`Elicitation` hook](/en/hooks#elicitation).
   957→
   958→If you're building an MCP server that uses elicitation, see the [MCP elicitation specification](https://modelcontextprotocol.io/docs/learn/client-concepts#elicitation) for protocol details and schema examples.

  1001→## Scale with MCP Tool Search
  1002→
  1003→When you have many MCP servers configured, tool definitions can consume a significant portion of your context window. MCP Tool Search solves this by dynamically loading tools on-demand instead of preloading all of them.
  1004→
  1005→### How it works
  1006→
  1007→Claude Code automatically enables Tool Search when your MCP tool descriptions would consume more than 10% of the context window. You can [adjust this threshold](#configure-tool-search) or disable tool search entirely. When triggered:
  1008→
  1009→1. MCP tools are deferred rather than loaded into context upfront
  1010→2. Claude uses a search tool to discover relevant MCP tools when needed
  1011→3. Only the tools Claude actually needs are loaded into context
  1012→4. MCP tools continue to work exactly as before from your perspective
  1013→
  1014→### For MCP server authors
  1015→
  1016→If you're building an MCP server, the server instructions field becomes more useful with Tool Search enabled. Server instructions help Claude understand when to search for your tools, similar to how [skills](/en/skills) work.
  1017→
  1018→Add clear, descriptive server instructions that explain:
  1019→
  1020→* What category of tasks your tools handle
  1021→* When Claude should search for your tools
  1022→* Key capabilities your server provides
  1023→
  1024→### Configure tool search
  1025→
  1026→Tool search is enabled by default: MCP tools are deferred and discovered on demand. When `ANTHROPIC_BASE_URL` points to a non-first-party host, tool search is disabled by default because most proxies do not forward `tool_reference` blocks. Set `ENABLE_TOOL_SEARCH` explicitly if your proxy does. This feature requires models that support `tool_reference` blocks: Sonnet 4 and later, or Opus 4 and later. Haiku models do not support tool search.
  1027→
  1028→Control tool search behavior with the `ENABLE_TOOL_SEARCH` environment variable:
  1029→
  1030→| Value      | Behavior                                                                           |
  1031→| :--------- | :--------------------------------------------------------------------------------- |
  1032→| (unset)    | Enabled by default. Disabled when `ANTHROPIC_BASE_URL` is a non-first-party host   |
  1033→| `true`     | Always enabled, including for non-first-party `ANTHROPIC_BASE_URL`                 |
  1034→| `auto`     | Activates when MCP tools exceed 10% of context                                     |
  1035→| `auto:<N>` | Activates at custom threshold, where `<N>` is a percentage (e.g., `auto:5` for 5%) |
  1036→| `false`    | Disabled, all MCP tools loaded upfront                                             |
  1037→
  1038→```bash  theme={null}
  1039→# Use a custom 5% threshold
  1040→ENABLE_TOOL_SEARCH=auto:5 claude
  1041→
  1042→# Disable tool search entirely
  1043→ENABLE_TOOL_SEARCH=false claude
  1044→```
  1045→
  1046→Or set the value in your [settings.json `env` field](/en/settings#available-settings).
  1047→
  1048→You can also disable the MCPSearch tool specifically using the `disallowedTools` setting:
  1049→
  1050→```json  theme={null}
  1051→{
  1052→  "permissions": {
  1053→    "deny": ["MCPSearch"]
  1054→  }
  1055→}
  1056→```

  1097→## Managed MCP configuration
  1098→
  1099→For organizations that need centralized control over MCP servers, Claude Code supports two configuration options:
  1100→
  1101→1. **Exclusive control with `managed-mcp.json`**: Deploy a fixed set of MCP servers that users cannot modify or extend
  1102→2. **Policy-based control with allowlists/denylists**: Allow users to add their own servers, but restrict which ones are permitted
  1103→
  1104→These options allow IT administrators to:
  1105→
  1106→* **Control which MCP servers employees can access**: Deploy a standardized set of approved MCP servers across the organization
  1107→* **Prevent unauthorized MCP servers**: Restrict users from adding unapproved MCP servers
  1108→* **Disable MCP entirely**: Remove MCP functionality completely if needed
  1109→
  1110→### Option 1: Exclusive control with managed-mcp.json
  1111→
  1112→When you deploy a `managed-mcp.json` file, it takes **exclusive control** over all MCP servers. Users cannot add, modify, or use any MCP servers other than those defined in this file. This is the simplest approach for organizations that want complete control.
  1113→
  1114→System administrators deploy the configuration file to a system-wide directory:
  1115→
  1116→* macOS: `/Library/Application Support/ClaudeCode/managed-mcp.json`
  1117→* Linux and WSL: `/etc/claude-code/managed-mcp.json`
  1118→* Windows: `C:\Program Files\ClaudeCode\managed-mcp.json`
  1119→
  1120→<Note>
  1121→  These are system-wide paths (not user home directories like `~/Library/...`) that require administrator privileges. They are designed to be deployed by IT administrators.
  1122→</Note>
  1123→
  1124→The `managed-mcp.json` file uses the same format as a standard `.mcp.json` file:
  1125→
  1126→```json  theme={null}
  1127→{
  1128→  "mcpServers": {
  1129→    "github": {
  1130→      "type": "http",
  1131→      "url": "https://api.githubcopilot.com/mcp/"
  1132→    },
  1133→    "sentry": {
  1134→      "type": "http",
  1135→      "url": "https://mcp.sentry.dev/mcp"
  1136→    },
  1137→    "company-internal": {
  1138→      "type": "stdio",
  1139→      "command": "/usr/local/bin/company-mcp-server",
  1140→      "args": ["--config", "/etc/company/mcp-config.json"],
  1141→      "env": {
  1142→        "COMPANY_API_URL": "https://internal.company.com"
  1143→      }
  1144→    }
  1145→  }
  1146→}
  1147→```
  1148→
  1149→### Option 2: Policy-based control with allowlists and denylists
  1150→
  1151→Instead of taking exclusive control, administrators can allow users to configure their own MCP servers while enforcing restrictions on which servers are permitted. This approach uses `allowedMcpServers` and `deniedMcpServers` in the [managed settings file](/en/settings#settings-files).
  1152→
  1153→<Note>
  1154→  **Choosing between options**: Use Option 1 (`managed-mcp.json`) when you want to deploy a fixed set of servers with no user customization. Use Option 2 (allowlists/denylists) when you want to allow users to add their own servers within policy constraints.
  1155→</Note>
  1156→
  1157→#### Restriction options
  1158→
  1159→Each entry in the allowlist or denylist can restrict servers in three ways:
  1160→
  1161→1. **By server name** (`serverName`): Matches the configured name of the server
  1162→2. **By command** (`serverCommand`): Matches the exact command and arguments used to start stdio servers
  1163→3. **By URL pattern** (`serverUrl`): Matches remote server URLs with wildcard support
  1164→
  1165→**Important**: Each entry must have exactly one of `serverName`, `serverCommand`, or `serverUrl`.
  1166→
  1167→#### Example configuration
  1168→
  1169→```json  theme={null}
  1170→{
  1171→  "allowedMcpServers": [
  1172→    // Allow by server name
  1173→    { "serverName": "github" },
  1174→    { "serverName": "sentry" },
  1175→
  1176→    // Allow by exact command (for stdio servers)
  1177→    { "serverCommand": ["npx", "-y", "@modelcontextprotocol/server-filesystem"] },
  1178→    { "serverCommand": ["python", "/usr/local/bin/approved-server.py"] },
  1179→
  1180→    // Allow by URL pattern (for remote servers)
  1181→    { "serverUrl": "https://mcp.company.com/*" },
  1182→    { "serverUrl": "https://*.internal.corp/*" }
  1183→  ],
  1184→  "deniedMcpServers": [
  1185→    // Block by server name
  1186→    { "serverName": "dangerous-server" },
  1187→
  1188→    // Block by exact command (for stdio servers)
  1189→    { "serverCommand": ["npx", "-y", "unapproved-package"] },
  1190→
  1191→    // Block by URL pattern (for remote servers)
  1192→    { "serverUrl": "https://*.untrusted.com/*" }
  1193→  ]
  1194→}
  1195→```
  1196→
  1197→#### How command-based restrictions work
  1198→
  1199→**Exact matching**:
  1200→
  1201→* Command arrays must match **exactly** - both the command and all arguments in the correct order
  1202→* Example: `["npx", "-y", "server"]` will NOT match `["npx", "server"]` or `["npx", "-y", "server", "--flag"]`
  1203→
  1204→**Stdio server behavior**:
  1205→
  1206→* When the allowlist contains **any** `serverCommand` entries, stdio servers **must** match one of those commands
  1207→* Stdio servers cannot pass by name alone when command restrictions are present
  1208→* This ensures administrators can enforce which commands are allowed to run
  1209→
  1210→**Non-stdio server behavior**:
  1211→
  1212→* Remote servers (HTTP, SSE, WebSocket) use URL-based matching when `serverUrl` entries exist in the allowlist
  1213→* If no URL entries exist, remote servers fall back to name-based matching
  1214→* Command restrictions do not apply to remote servers
  1215→
  1216→#### How URL-based restrictions work
  1217→
  1218→URL patterns support wildcards using `*` to match any sequence of characters. This is useful for allowing entire domains or subdomains.
  1219→
  1220→**Wildcard examples**:
  1221→
  1222→* `https://mcp.company.com/*` - Allow all paths on a specific domain
  1223→* `https://*.example.com/*` - Allow any subdomain of example.com
  1224→* `http://localhost:*/*` - Allow any port on localhost
  1225→
  1226→**Remote server behavior**:
  1227→
  1228→* When the allowlist contains **any** `serverUrl` entries, remote servers **must** match one of those URL patterns
  1229→* Remote servers cannot pass by name alone when URL restrictions are present
  1230→* This ensures administrators can enforce which remote endpoints are allowed
  1231→
  1232→<Accordion title="Example: URL-only allowlist">
  1233→  ```json  theme={null}
  1234→  {
  1235→    "allowedMcpServers": [
  1236→      { "serverUrl": "https://mcp.company.com/*" },
  1237→      { "serverUrl": "https://*.internal.corp/*" }
  1238→    ]
  1239→  }
  1240→  ```
  1241→
  1242→  **Result**:
  1243→
  1244→  * HTTP server at `https://mcp.company.com/api`: ✅ Allowed (matches URL pattern)
  1245→  * HTTP server at `https://api.internal.corp/mcp`: ✅ Allowed (matches wildcard subdomain)
  1246→  * HTTP server at `https://external.com/mcp`: ❌ Blocked (doesn't match any URL pattern)
  1247→  * Stdio server with any command: ❌ Blocked (no name or command entries to match)
  1248→</Accordion>
  1249→
  1250→<Accordion title="Example: Command-only allowlist">
  1251→  ```json  theme={null}
  1252→  {
  1253→    "allowedMcpServers": [
  1254→      { "serverCommand": ["npx", "-y", "approved-package"] }
  1255→    ]
  1256→  }
  1257→  ```
  1258→
  1259→  **Result**:
  1260→
  1261→  * Stdio server with `["npx", "-y", "approved-package"]`: ✅ Allowed (matches command)
  1262→  * Stdio server with `["node", "server.js"]`: ❌ Blocked (doesn't match command)
  1263→  * HTTP server named "my-api": ❌ Blocked (no name entries to match)
  1264→</Accordion>
  1265→
  1266→<Accordion title="Example: Mixed name and command allowlist">
  1267→  ```json  theme={null}
  1268→  {
  1269→    "allowedMcpServers": [
  1270→      { "serverName": "github" },
  1271→      { "serverCommand": ["npx", "-y", "approved-package"] }
  1272→    ]
  1273→  }
  1274→  ```
  1275→
  1276→  **Result**:
  1277→
  1278→  * Stdio server named "local-tool" with `["npx", "-y", "approved-package"]`: ✅ Allowed (matches command)
  1279→  * Stdio server named "local-tool" with `["node", "server.js"]`: ❌ Blocked (command entries exist but doesn't match)
  1280→  * Stdio server named "github" with `["node", "server.js"]`: ❌ Blocked (stdio servers must match commands when command entries exist)
  1281→  * HTTP server named "github": ✅ Allowed (matches name)
  1282→  * HTTP server named "other-api": ❌ Blocked (name doesn't match)
  1283→</Accordion>
  1284→
  1285→<Accordion title="Example: Name-only allowlist">
  1286→  ```json  theme={null}
  1287→  {
  1288→    "allowedMcpServers": [
  1289→      { "serverName": "github" },
  1290→      { "serverName": "internal-tool" }
  1291→    ]
  1292→  }
  1293→  ```
  1294→
  1295→  **Result**:
  1296→
  1297→  * Stdio server named "github" with any command: ✅ Allowed (no command restrictions)
  1298→  * Stdio server named "internal-tool" with any command: ✅ Allowed (no command restrictions)
  1299→  * HTTP server named "github": ✅ Allowed (matches name)
  1300→  * Any server named "other": ❌ Blocked (name doesn't match)
  1301→</Accordion>
  1302→
  1303→#### Allowlist behavior (`allowedMcpServers`)
  1304→
  1305→* `undefined` (default): No restrictions - users can configure any MCP server
  1306→* Empty array `[]`: Complete lockdown - users cannot configure any MCP servers
  1307→* List of entries: Users can only configure servers that match by name, command, or URL pattern
  1308→
  1309→#### Denylist behavior (`deniedMcpServers`)
  1310→
  1311→* `undefined` (default): No servers are blocked
  1312→* Empty array `[]`: No servers are blocked
  1313→* List of entries: Specified servers are explicitly blocked across all scopes
  1314→
  1315→#### Important notes
  1316→
  1317→* **Option 1 and Option 2 can be combined**: If `managed-mcp.json` exists, it has exclusive control and users cannot add servers. Allowlists/denylists still apply to the managed servers themselves.
  1318→* **Denylist takes absolute precedence**: If a server matches a denylist entry (by name, command, or URL), it will be blocked even if it's on the allowlist
  1319→* Name-based, command-based, and URL-based restrictions work together: a server passes if it matches **either** a name entry, a command entry, or a URL pattern (unless blocked by denylist)
  1320→
  1321→<Note>
  1322→  **When using `managed-mcp.json`**: Users cannot add MCP servers through `claude mcp add` or configuration files. The `allowedMcpServers` and `deniedMcpServers` settings still apply to filter which managed servers are actually loaded.
  1323→</Note>
