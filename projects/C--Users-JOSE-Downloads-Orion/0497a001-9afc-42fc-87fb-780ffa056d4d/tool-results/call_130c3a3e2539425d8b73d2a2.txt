    22→## What Changed
    23→
    24→v2.4 introduced The Algorithm—a 7-phase approach to problem-solving with verifiable criteria. It worked, but it had blind spots: it picked capabilities based on habit rather than analysis, it skipped thinking tools without justification, and it ran independent tasks one at a time when they could have run simultaneously.
    25→
    26→v2.5 fixes all three.
    27→
    28→### The Problem v2.5 Solves
    29→
    30→v2.4's Algorithm:
    31→```
    32→Pick familiar tools → Execute sequentially → Hope the right capabilities were chosen
    33→```
    34→
    35→v2.5's Algorithm:
    36→```
    37→Hook suggests capabilities → Reverse-engineer true intent → Validate suggestions against ISC →
    38→Justify every tool inclusion/exclusion → Execute independent work in parallel → Verify
    39→```
    40→
    41→The difference is metacognition. The Algorithm now thinks about *how* it thinks before it acts.
    42→

    44→
    45→## Three Major Features
    46→
    47→### 1. Two-Pass Capability Selection
    48→
    49→Previously, capability selection happened implicitly—the system defaulted to familiar patterns (usually Engineer + Research) regardless of what the task actually needed.
    50→
    51→Now selection happens in two explicit passes:
    52→
    53→**Pass 1 — Hook Hints (before the Algorithm starts)**
    54→
    55→The FormatReminder hook runs AI inference on the raw prompt and suggests capabilities, skills, and thinking tools. These are draft suggestions—a head start, not a decision.
    56→
    57→**Pass 2 — THINK Validation (after OBSERVE completes)**
    58→
    59→With the full context of reverse-engineering and ISC criteria, the THINK phase validates every suggestion. Skills get added, removed, or confirmed. Pass 2 is authoritative.
    60→
    61→```
    62→🔍 SKILL CHECK (validate hook hints against ISC):
    63→│ Hook suggested:   CreateSkill:UpdateSkill
    64→│ ISC requires:     Architecture design (not just skill update)
    65→│ Final skills:     CreateSkill:UpdateSkill + Architect consultation
    66→```
    67→
    68→**Why two passes?** The hook gives a head start from the raw prompt. But OBSERVE changes the picture. A request that looks like a simple skill update might actually require architectural decisions, or might rest on questionable assumptions that need First Principles analysis. Pass 2 catches what Pass 1 cannot see.
    69→
    70→---
    71→
    72→### 2. Thinking Tools with Justify-Exclusion
    73→
    74→v2.4 had thinking tools (Council, RedTeam, FirstPrinciples, etc.) but they were rarely used because the default was to skip them. v2.5 inverts this: **thinking tools are opt-OUT, not opt-IN.**
    75→
    76→For every request, the Algorithm must evaluate each thinking tool and justify why it is NOT being used:
    77→
    78→```
    79→🔍 THINKING TOOLS ASSESSMENT (justify exclusion):
    80→│ Council:          EXCLUDE — single clear approach, no alternatives to debate
    81→│ RedTeam:          INCLUDE — proposal could fail in non-obvious ways
    82→│ FirstPrinciples:  INCLUDE — requirement rests on unexamined assumption
    83→│ Science:          EXCLUDE — not iterative/experimental
    84→│ BeCreative:       EXCLUDE — clear requirements, no divergence needed
    85→│ Prompting:        EXCLUDE — not generating prompts
    86→```
    87→
    88→The burden of proof is on exclusion. "Too simple" and "already know the answer" are not valid reasons—simple tasks can have hidden assumptions, and confidence without verification is exactly the failure mode this catches.
    89→
    90→### The Six Thinking Tools
    91→
    92→| Tool | What It Does | Include When |
    93→|------|-------------|--------------|
    94→| **Council** | Multi-agent debate with 3-7 agents | Multiple valid approaches exist |
    95→| **RedTeam** | Adversarial analysis with 32 agents | Claims need stress-testing |
    96→| **FirstPrinciples** | Deconstruct, challenge, reconstruct | Assumptions need examining |
    97→| **Science** | Hypothesis-test-analyze cycles | Iterative experimentation needed |
    98→| **BeCreative** | Extended thinking, 5 diverse options | Creative divergence needed |
    99→| **Prompting** | Meta-prompting with templates | Prompt optimization needed |
   100→
   101→---
   102→
   103→### 3. Parallel-by-Default Execution
   104→
   105→When the Algorithm identifies independent tasks during BUILD/EXECUTE, they now launch simultaneously as concurrent agents in a single message. Serial execution of independent tasks is a failure mode.
   106→
   107→```
   108→BEFORE (v2.4):          AFTER (v2.5):
   109→Task A → wait           Task A ─┐
   110→Task B → wait           Task B ─┼─ all at once
   111→Task C → wait           Task C ─┘
   112→Task D → done           Task D → done
   113→```
   114→
   115→**The rule is simple:** If tasks don't depend on each other's output, they run at the same time. Period.
   116→
   117→This applies automatically to:
   118→- Multiple file edits with no cross-dependencies
   119→- Multiple research queries on different topics
   120→- Multiple audits or scans of independent systems
   121→- Multiple creation tasks with no shared state
   122→
   123→Fan-out is the default pattern for 3+ independent workstreams.

   127→## Composition Patterns
   128→
   129→Capabilities now combine using 7 named patterns, making orchestration explicit and visible:
   130→
   131→| Pattern | Shape | Example | When |
   132→|---------|-------|---------|------|
   133→| **Pipeline** | A → B → C | Explore → Architect → Engineer | Sequential domain handoff |
   134→| **TDD Loop** | A ↔ B | Engineer ↔ QA | Build-verify cycle until ISC passes |
   135→| **Fan-out** | → [A, B, C] | 3 Researchers in parallel | Multiple perspectives needed |
   136→| **Fan-in** | [A, B, C] → D | Researchers → Synthesis | Merging parallel results |
   137→| **Gate** | A → check → B or retry | Engineer → QA → Deploy or fix | Quality gate before progression |
   138→| **Escalation** | A(haiku) → A(sonnet) → A(opus) | Model upgrade on failure | Complexity exceeded model tier |
   139→| **Specialist** | Single A | Pentester for security review | One domain, deep expertise |
   140→
   141→The full Capability Selection block in THINK phase now looks like:
   142→
   143→```
   144→🎯 CAPABILITY SELECTION:
   145→│ Skills:     CreateSkill:UpdateSkill
   146→│ Thinking:   FirstPrinciples, RedTeam
   147→│ Primary:    Architect  — system design decision (ISC #1)
   148→│ Support:    Engineer   — implementation (ISC #2-4)
   149→│ Verify:     QATester   — browser verification (ISC #5)
   150→│ Pattern:    Pipeline
   151→│ Sequence:   Architect → Engineer → QATester
   152→│ Rationale:  Architecture must be decided before implementation begins
   153→```

   227→## Quick Start
   228→
   229→```bash
   230→# 1. Clone the repo (if you haven't already)
   231→git clone https://github.com/danielmiessler/PAI.git
   232→cd PAI/Releases/v2.5
   233→
   234→# 2. Backup existing installation (if any)
   235→[ -d ~/.claude ] && mv ~/.claude ~/.claude-backup-$(date +%Y%m%d)
   236→
   237→# 3. Copy the complete release
   238→cp -r .claude ~/
   239→
   240→# 4. Run the configuration wizard
   241→cd ~/.claude && bun run INSTALL.ts
   242→
   243→# 5. Restart Claude Code to activate hooks

   337→## The Goal: Euphoric Surprise
   338→
   339→The target remains **Euphoric Surprise**—results so thorough you're genuinely delighted, not just satisfied.
   340→
   341→v2.5 makes this more achievable by addressing three weaknesses in v2.4:
   342→
   343→1. **Better tool selection** — Two-pass validation means the right capabilities are chosen based on evidence, not habit
   344→2. **Better metacognition** — Thinking tools catch hidden assumptions and unexplored alternatives
   345→3. **Better throughput** — Parallel execution means complex tasks complete faster without sacrificing quality
   346→
   347→The Algorithm's hill-climbing loop (define criteria → execute → verify → learn) now has sharper tools at every stage.
